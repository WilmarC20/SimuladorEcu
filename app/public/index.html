Emulador GPS NEO

        

          Pines que usa el emulador:

          GPIO14 (pin 8) = TX · GPIO15 (pin 10) = RX · GND = pin 6 o 9. Salida 3,3 V. Conectar TX de la Pi → RX del dispositivo que quieres que reciba el GPS (como un NEO real).
        

        

          

            
Cómo simular el GPS (paso a paso)

            
    
              
  1. Dejar libre el UART (solo la primera vez). En la Pi ejecuta sudo raspi-config → Interface Options → Serial Port. Pregunta “Login shell over serial”: No. Pregunta “Serial port hardware enabled”: Sí. Reinicia. Así el puerto /dev/ttyS0 (o /dev/serial0) lo usa el dashboard y no la consola.
    2. 
              
  2. En esta página (Emulador GPS NEO): En el desplegable elige el puerto (/dev/ttyS0 o /dev/serial0 para pines GPIO). Baud rate suele ser 9600. Pulsa Abrir puerto. Debe aparecer “Abierto: /dev/...”.
    3. 
              
  3. Configurar y arrancar el envío: En esta misma página, en la sección Posición y envío por serial (más abajo), puedes:
                
      
                  
    • Poner lat, lon (y opcional alt, satélites, rumbo) y dejar “ruta off” para un punto fijo.
      • 
                  
    • O activar “ruta on”, poner radio (m) y vel km/h para simular un recorrido en círculo.
      • 
                
    
                Pulsa Aplicar y luego Iniciar envío por serial. El dashboard enviará tramas NMEA (GPRMC, GPGGA, etc.) por el pin TX de la Pi cada cierto intervalo (campo “ms”).
    4. 
              
  4. El dispositivo conectado a los pines (su RX al TX de la Pi, GND común) recibirá esas tramas como si fuera un GPS NEO real.
    5. 
            

            
Si no ves el puerto en el desplegable, recarga la página. Si al abrir da error, comprueba que la consola serial esté desactivada en raspi-config.

Información que envía el GPS

            
Por el pin TX (GPIO 14) se envían tramas NMEA 0183 cada cierto intervalo (campo «ms»):

            
    
              
  • $GPGGA — Hora UTC, latitud, longitud, calidad del fix (0/1), número de satélites, HDOP, altitud (m).
    • 
              
  • $GPRMC — Hora UTC, estado (A=válido, V=inválido), latitud, longitud, velocidad (nudos), rumbo (°), fecha UTC.
    • 
            

            
Cada trama termina en *XX (checksum). El dispositivo conectado al RX recibe estas líneas como un GPS NEO real.

            
            
Abre el puerto serial arriba y pulsa Iniciar envío por serial para ver las tramas aquí.

      Conexión OBD2 hembra a la Raspberry Pi 4
      
            
      Dos opciones habituales:
      
            
      
              
      
                
      Opción A — Adaptador USB-OBD2
      
                
      Cable/adaptador con conector hembra OBD2 en un extremo y USB en el otro. Se conecta a un puerto USB de la Pi. En la Pi suele verse como /dev/ttyUSB0.

      3. Ver tráfico CAN (candump)
      
            
      Activar SPI y overlay MCP2515 en /boot/firmware/config.txt (o /boot/config.txt):
      
            
      dtparam=spi=on
dtoverlay=mcp2515-can0,oscillator=8000000,interrupt=25
      
            
      Usa oscillator=8000000 si el cristal del MCP2515 es 8 MHz (ej. dice "8.000"). Si es 16 MHz, pon 16000000.
      
            
      Herramientas: sudo apt install -y can-utils. Subir interfaz y ver tráfico:
      
            
      sudo ip link set can0 up type can bitrate 500000
candump can0
      
            
      O ejecutar sudo ~/dashboard/scripts/can0-up.sh para subir can0.
      
            
      4. Arranque al reiniciar la Pi (servicio)
      
            
      Para que el bus CAN esté UP antes de que arranque la app (y el kiosk):
      
            
        
              
      • Script scripts/ensure-can0-boot.sh: espera hasta 20 s a que exista can0 (MCP2515) y lo sube a 500 kbps.
        • 
              
      • Servicio tracker360-dashboard.service: ejecuta ese script y luego la app. Instalación: sudo cp deploy/tracker360-dashboard.service /etc/systemd/system/, sudo systemctl daemon-reload, sudo systemctl enable tracker360-dashboard.service.
        • 
              
      • Si al ejecutar los scripts sale "No such file", suele ser por finales de línea Windows (CRLF). En la Pi: sed -i 's/\r$//' ~/dashboard/scripts/ensure-can0-boot.sh ~/dashboard/scripts/can0-up.sh y chmod +x ~/dashboard/scripts/*.sh.
        • 
            
      
            
      5. Terminación del bus (J1)
      
            
      Muchos módulos MCP2515 tienen un jumper J1 que activa una resistencia de 120 Ω. Con dos nodos en el bus (Pi + lector/ELM) suele ir sin el jumper en la Pi si el otro nodo ya lleva terminación; si no hay comunicación, prueba con J1 puesto o quitado.

      7. Solución de problemas
      
            
      
              
      No aparece «Conectado al emulador OBD2»
      
              
      Compruebe que el servidor esté en marcha y que entre con la IP correcta y puerto 3000. Recargue (F5).
      
              
      Torque (o la app) no conecta por WiFi
      
              
      Use la IP de la Pi y puerto 35000. Celular y Pi en la misma red WiFi.
      
              
      Bluetooth: error al emparejar
      
              
      En la Pi instale python3-dbus y python3-gi. En Sistema pulse «Ver log del agente».
      
              
      El GPS no envía datos
      
              
      1) Abra el puerto en Emulador GPS NEO. 2) En OBD2 → GPS pulse Aplicar e Iniciar envío por serial. 3) UART habilitado en raspi-config, consola serial desactivada.
      
              
      El mapa no carga o sale «Esta página no puede cargar Google Maps correctamente»
      
              
      1) Tenga GOOGLE_MAPS_API_KEY en .env. 2) En Google Cloud Console → Credentials pulse su clave API → Restricciones de aplicación: si usa «Referentes HTTP», añada http://192.168.0.64:3000/* y http://localhost:3000/* (o la IP/puerto desde la que abre el dashboard). 3) En Restricciones de API permita «Maps JavaScript API» y «Directions API» (o «No restringir clave» para pruebas). 4) El proyecto debe tener facturación habilitada (Google da crédito gratuito).
      
            
      
          
      
        
      
      
      
      
      
        
      Importar exportación de Torque Pro
      
        
      Sube el archivo ZIP que exportaste desde Torque (Trip Logs Export). Se mostrará un listado por carpetas para que elijas qué partes importar.
      
        
      
          
      
            
      ¿Para qué sirve cada tipo de archivo?
      
            
      En el ZIP de Torque suelen venir dos subcarpetas y archivos con estos formatos:
      
            
      
              
      .csv
      
              
      Registros de viaje (trip logs): cada fila es un instante con tiempo, velocidad (OBD), RPM, temperatura, latitud, longitud, altitud GPS, etc. Sirven para reproducir el trayecto en el emulador, extraer rutas o analizar datos.
      
              
      .log
      
              
      Log en bruto del viaje (a veces el mismo contenido que el CSV con otro nombre, o formato interno de Torque). Útil como respaldo o para herramientas que lean ese formato.
      
              
      Subcarpetas (p. ej. por fecha o por viaje)
      
              
      Torque puede agrupar los archivos en carpetas (por día, por sesión, etc.). Puedes importar una carpeta entera o solo los archivos que te interesen.
      
            
      
            
      Todo lo que importes se guarda en app/data/torque-logs/. Esos CSV se usan en el Logger del Emulador OBD2 (reproducción con sensores).
